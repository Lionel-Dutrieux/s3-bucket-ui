# Adoption Quick Wins — Design

**Date**: 2026-07-17
**Status**: Approved (sections 1, 2, 3, 5 — live demo deferred)
**Goal**: Increase adoption / GitHub visibility of Bucket UI without adding
product features, by sharpening positioning and lowering the discovery +
install barrier. No competitor is ever named.

## Context

Bucket UI is a mature self-hosted app (auth/grants/audit, 13 object stores +
SFTP/FTP/WebDAV, sharing, ZIP, cross-provider copy, i18n ×5, multi-arch image,
healthcheck). It is no longer feature-limited; it is **discovery-limited**. The
README has no screenshots, there is no live demo, and it is absent from the
catalogs and lists the self-hosted community browses.

This chantier is a bundle of adoption "quick wins" (~1–2 evenings), scoped so
that most deliverables are producible without the user, and screenshots are
handled in a later pass.

## Positioning spine (the one pitch, reused everywhere)

Every deliverable below draws from the same pitch. **Rule: never name another
open-source or commercial solution.** The framing is problem-first, not
feature-list-first.

- **Problem**: giving someone access to a bucket today means handing them the
  S3 credentials. One console per provider, built for engineers.
- **Bucket UI**: one interface for every store you own (object stores +
  protocols), with per-user / per-group access enforced server-side. It
  **grants access to an S3 bucket without ever handing out its credentials.**
  Enterprise-grade access control — native OIDC SSO, grants, audit trail,
  white-labeling — **free and self-hosted.**
- **Trust line**: the database only holds authentication, permissions and
  source definitions. **Every file operation goes straight to your storage via
  file-SDK — Bucket UI never copies or stores your files.** (No lock-in, no
  data duplication.)
- **Origin story (sidebar/quote)**: "I built Bucket UI to manage the S3
  buckets behind my Dokploy apps — pulling backups, curating an app's media,
  handing a single document to someone who had no bucket access."

Tone: confident, practical, homelab-and-enterprise-friendly. Emphasis order:
(1) access without credentials, (2) enterprise security free, (3) no-lock-in.

## Deliverables

### 1. README refresh — *fully by Claude*

Edit `README.md`:

- Rewrite the **"Why Bucket UI?"** section using the positioning spine, with
  the origin story as a short blockquote/callout.
- Add a concise **"How it works"** block: DB = auth / permissions / sources;
  files = direct to storage via file-SDK. State the no-copy / no-lock-in
  guarantee explicitly.
- Insert placeholders for a **hero screenshot** (near the top, under the title)
  and a **demo GIF** (in Features), as HTML comments carrying the exact
  shot-list, e.g. `<!-- hero: list view, grid with thumbnails, breadcrumbs -->`.
  Placeholders must not break rendering (no broken `<img>` — use commented-out
  markup ready to uncomment once assets exist).
- Leave Quick start, Configuration, Supported storage, Security model, Tech
  stack untouched — they are already strong.

Constraint: keep it scannable; the "Why" + "How it works" together stay under
~250 words so the value lands above the fold.

### 2. Dokploy template — *fully by Claude*

Produce a template for the `Dokploy/templates` repo (dogfooding — the tool was
built for Dokploy). Use the `dokploy-toolkit:dokploy-template` skill.

- Files: `docker-compose.yml`, `template.toml`, `meta.json` in the Dokploy
  template layout.
- Base it on `docker-compose.demo.yml` (app + bundled PostgreSQL).
- Secrets (`ENCRYPTION_KEY` via `openssl rand -hex 32`, `BETTER_AUTH_SECRET`
  via `openssl rand -base64 32`) generated by the template; domain + `BETTER_AUTH_URL`
  wired through the Dokploy UI variables.
- Healthcheck already present (`/api/health`) — surface it.
- Deliverable placed under `docs/` or a dedicated folder in this repo, ready to
  copy into a `Dokploy/templates` PR. Do **not** open the upstream PR
  automatically — leave that to the user.

### 3. awesome-selfhosted entry + launch post — *fully by Claude*

- **awesome-selfhosted entry**: exact list format (name, ≤250-char description
  drawn from the pitch, source URL, demo, license `MIT`, language `Nodejs`).
  Verify current formatting rules against the live `awesome-selfhosted`
  contributing guide before writing. Provide the one-line entry + which
  category section it belongs in + a ready-to-paste PR description. User opens
  the PR.
- **r/selfhosted launch post draft**: title + body that lead with the problem
  (access without credentials) rather than a feature dump; include a spot for
  the GIF/screenshots and the repo link. Saved as a markdown file in the repo
  (e.g. `docs/launch/`). User publishes on their own timing.

## Execution order

1 (positioning is baked into 2/3) → README → Dokploy template →
awesome-selfhosted + Reddit draft.

## Out of scope (follow-ups, not this chantier)

- **Live public demo** (#4 from brainstorming) — deferred: abuse surface +
  maintenance. Revisit after screenshots exist.
- Screenshots / demo GIF capture — a later pass; this chantier only prepares
  the slots and the shot-list.
- Feature work: PWA/mobile, ntfy/Apprise notifications, Prometheus metrics,
  WebDAV-server export. These are "feature" chantiers, not adoption quick wins.

## Success criteria

- README communicates the "access without credentials + enterprise-free +
  no-lock-in" pitch above the fold, with visual slots ready.
- A Dokploy template exists in-repo, valid, ready for an upstream PR.
- An awesome-selfhosted entry + a launch-post draft exist, both derived from
  the single pitch, both ready for the user to submit/publish.
- No competitor named anywhere.
